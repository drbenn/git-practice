Managing Branches:

  1. create branch with 'git branch feature1'
  2. switch between branches with 'git switch feature1'
  3. to delete branch, must not be in branch being deleted and use 'git branch -D feature1', the big -D is like --force and there is no coming back after
  4. To rename use -m which is for move/rename, you MUST be in the branch you are renaming and then do 'git branch -m new-name', where new name is the replacement branch name. May need change in code for suiteable commit and then an additional command of 'git push origin HEAD' as gitHub will not recognize the new branch name

For simple merge (feature branch with no addtl commits on master branch):

  1. switch to branch receiving the feature branch, in this case, 'git switch main'
  2. merge feature branch INTO master branch, 'git merge feature1'... i did 'git merge renamed-branch'
  3. In the case for ex. branch 'SeachBar' was created with several commits, and after it was branched off 'main' branch added a few commits, however, updates to totally different files were complete in each branch, there will be no merge conflicts to validate, however, a merge comment will be necessary

For Merge Conflicts:

  1. Open files with merge conflicts
  2. Edit files to remove conflicts, content from both branches will be shown, you need to keep/replace what is necessary
  3. Remove the conflict 'markers' in the doc
  4. Add your changes and then make a commit
  • VSC can be used to select one branch or the other with clickable options to select which branches edits to accept in the merge

Git Stash - How to save uncommitted edits on new branch and then switch branch(bc cant switch branch bc uncommitted!)

  • If there are NO conflicts between moving branches, the uncommitted work will follow you to the switched to branch
  • if there ARE conflicts, you will need to delete changes or you can stash
    1. If encounter this uncommitted conflict, just use 'git stash' (short for git stash save)
    2. Then later use 'git stash pop' to reapply the stashed changes to your working copy...which CAN be a different branch

Rebasing

Two main ways to use rebase

  1. Alternate to merge
  2. As a cleanup tool
  3. NEVER REBASE commits that have been shared with others(others created a branch off your feature branch)
  • Rebase adjusts the timeline and re-writes history.
  • Instead of parallel commits between two branches, if you rebased a feature branch onto main, the feature commits would all be added to extend the HEAD(at the tip) of the main branch
  • Instead of using a merge commit, rebasing rewrites history by creating new commits(after main HEAD) for each of the original feature branch commtis
  • 'git switch feature' and 'git rebase main' - you switch to branch merging in and you rebase the featuree into main branch
